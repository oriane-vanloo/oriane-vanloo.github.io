The problem with design exploration

Early-stage design exploration is time-consuming in ways that don't add value. When a team wants to explore three different directions for a UI pattern, the expensive part isn't the thinking. it's the execution: opening Figma, building each variant from scratch, making them all look presentable enough to evaluate and compare.

At Rome2Rio, the design team is small. Time spent on low-value execution is time not spent on decisions, craft and the problems only designers can solve. I wanted to change that ratio.

What I built (using Claude Code and Codex)

Custom AI skills. reusable, prompt-driven tools built on top of Claude Code. that generate multiple fully-rendered interactive UI variants from a written design brief. A designer writes what they're exploring and why, and gets back a set of working, comparable variants in the browser within minutes.

The variants are not static images. They're interactive HTML/CSS components, built to Hitch 2.0 design system standards. real enough to evaluate interaction behaviour, not just appearance.

1
Brief input The designer writes a brief: component type, user goal, design constraints, and which directions they want to explore. Plain language. no code required.
2
Variant generation The AI skill generates multiple distinct variants. each with a different design rationale. rendered as interactive components in the Hitch 2.0 visual language.
3
Side-by-side comparison Variants are presented in a notebook format: each direction visible alongside its rationale and trade-offs, making it fast to review, discuss and decide.
4
Iteration The designer can request refinements to any variant, push in a new direction, or combine elements from multiple. all through natural language, without touching code.

"The goal was never to replace design thinking. It was to remove the cost of acting on it."

Why this is a design process innovation

Traditional design tools are great at executing decisions. They're not optimised for generating options cheaply. AI changes that ratio. but only if the prompting layer is thoughtful enough to produce outputs worth evaluating.

The skill I built (using Claude Code and Codex) encodes design system knowledge, component constraints and the Hitch brand into the generation process. The result is variants that are already on-brand and realistic, not generic AI outputs that need extensive cleanup. The design team gets real creative exploration, not a starting point that needs as much work as building from scratch.

This project also represents a broader shift in how I think about my role as a principal designer: not just making better products, but building better ways for the team to make products.
